# AI时代算法专家成长计划 —— 项目制学习路线图

## 用户画像
- **当前能力**：后端程序员（Java/Python/前端），具备工程实践经验
- **架构水平**：初级架构师，能设计复杂架构、技术选型
- **目标**：升级为算法专家（工程型算法专家路线）
- **核心约束**：AI-Native学习、项目制驱动、非传统一步一步模式

## 核心设计理念

### 1. 为什么走"工程型算法专家"路线
传统算法研究员路线（发论文、刷竞赛）不适合有工程背景的你。工程型算法专家是2025-2026年最稀缺的T型人才：纵向有算法深度，横向有工程广度。企业需要的不是会背公式的人，而是能把算法落地成系统的工程师。

**专家标准的底线**：不能只会调API。必须理解模型权重的变化、数据流的每一步、硬件的每一个瓶颈。Prompt Engineering 和 RAG 只是表层，真正值钱的技能是——当你遇到一个"所有现成方案都不管用"的场景时，你能从数据配比、Loss曲线、内存布局、I/O吞吐的层面找到根因并给出方案。

### 2. AI-Native学习意味着什么
- **不徒手推导公式**：用AI解释算法直觉，快速建立认知框架
- **不从零写基础代码**：用AI生成脚手架，聚焦核心逻辑
- **不做无意义的练习题**：每个知识点锚定到项目中的具体位置
- **用AI当你的助教**：GPT/Claude是你的24小时导师，负责解释、答疑、代码review
- **但绝不只停留在API调用层**：每个项目都要深入到"如果API不可用，我能不能自己实现"的程度

### 3. 项目制学习的飞轮效应
每个项目 = 极端真实的业务场景 + 算法深度挑战 + 工程落地 + 可量化的交付物。9个项目构成一个完整的作品集，直接写进简历、在面试中深入讨论。

### 4. 本计划的四个硬核升级维度
| 维度 | 原计划的问题 | 升级后的要求 |
|------|-------------|-------------|
| **模型微调（SFT/RLHF）** | 前5个项目重度依赖开箱即用 | 新增项目7：LoRA/QLoRA领域微调，从数据配比到Loss收敛 |
| **真实复杂数据** | MovieLens/Yahoo S5太干净 | 项目3/4改用企业级招投标文档+复杂采购流程 |
| **多模态能力** | 100%纯文本/NLP | 项目8改为VLM+生成式图像工作流 |
| **底层硬件性能** | 只提到vLLM+A100 | 项目2/6/7深化到内存调度、GGUF/MLX、统一内存架构 |

### 项目1：智能代码审查助手（热身项目，2周）

**目标**：利用你的工程背景，快速建立"用算法解决工程问题"的信心。不追求复杂ML，而是用经典算法 + 规则实现实用功能。

**核心算法/技术**：
- 抽象语法树（AST）解析与遍历
- 代码相似度检测（编辑距离、最长公共子序列）
- 规则引擎设计（前向链推理）
- 代码复杂度分析（圈复杂度计算）

**AI辅助学习策略**：
- 让AI用Java/Python分别生成AST解析的对比实现，你负责看懂和整合
- 用AI生成测试用例，覆盖边界情况
- 让AI解释"为什么圈复杂度超过10就要告警"的工程直觉

**项目交付物**：
1. 一个能检测代码坏味道（长方法、重复代码、复杂条件）的命令行工具
2. 支持Java和Python双语言
3. 输出格式化的审查报告（JSON/Markdown）
4. 代码库集成方案（Git Hook/CI Pipeline）

**验证标准**：
- 能在一个真实开源项目（如Spring Boot Starter）上运行并发现至少10个有效问题
- 处理1000行代码的耗时 < 2秒
- 误报率 < 30%

**为什么这个项目**：
- 发挥你现有的工程优势，降低入门心理门槛
- 经典算法（树遍历、动态规划）是后续所有AI算法的基石
- 产出可以直接用在日常工作中，立即产生价值

### 项目2：高性能向量检索引擎（基础项目，4周）

**目标**：深入理解向量检索的核心算法和底层内存/I/O调度。从零实现一个简化版向量数据库，理解HNSW索引原理，并在"本地优先（Local-First）"架构下榨干硬件性能。

**核心算法/技术**：
- Embedding模型（Sentence-BERT/M3E）原理与使用
- 近似最近邻搜索（ANN）：LSH、IVF、HNSW
- HNSW图索引的构建与查询算法（多层导航图、贪心路由）
- 余弦相似度、点积、欧氏距离的数学本质与工程选择
- 向量量化（PQ/SQ）原理与内存压缩
- **底层I/O与内存调度**：mmap、零拷贝、SIMD向量化加速
- **统一内存架构（Apple Silicon）优化**：MLX框架、GGUF格式模型加载

**AI辅助学习策略**：
- 让AI用动画+伪代码解释HNSW的"多层高速公路"直觉
- 让AI生成HNSW的核心代码骨架，你负责实现关键的数据结构操作
- 用AI对比不同距离度量在不同场景下的表现
- 让AI帮你分析Milvus/Faiss的源码结构（不求读懂每一行，但求理解架构）
- **新增**：让AI帮你profiling内存访问模式（cacheline miss、page fault），找到真正的瓶颈

**项目交付物**：
1. 一个纯Python实现的简化版HNSW索引（支持增删查）
2. 一个C++/Rust版本的性能对比实现（使用SIMD优化）
3. Benchmark报告：与Faiss/HNSWlib的性能对比 + 内存/CPU profiling
4. **Local-First场景报告**：同一模型在Apple Silicon（MLX/GGUF）vs NVIDIA GPU（CUDA）下的性能差异分析
5. 技术博客：《从零实现HNSW：向量检索引擎的核心原理与内存优化》
6. 部署方案：Docker容器化 + REST API服务

**验证标准**：
- 在100万条128维向量上，查询延迟 < 10ms（召回率@100 > 95%）
- 索引构建内存占用 < 2GB
- 支持增量插入（不要求删除）
- 提供完整的性能评测报告，含CPU cache miss率和内存带宽利用率分析

**与工程架构的衔接**：
- 向量检索是现代AI系统的"记忆层"
- 理解索引结构+内存布局帮助你后续做RAG系统、推荐系统的技术选型
- 这个项目让你理解"算法复杂度 vs 工程性能"的trade-off——不只是Big-O，而是"数据在内存中的位置决定了速度"

**为什么增加I/O和硬件维度**：
- 专家与普通开发者的分水岭：前者知道数据从磁盘到CPU cache到寄存器的每一步延迟
- Local-First架构越来越重要（Apple Silicon的MLX、GGUF的本地推理）
- 理解这些让你在面试中能回答"为什么同样的算法在不同机器上差10倍"

### 项目3：复杂文档RAG系统（基础项目，4周）

**目标**：构建一个能处理企业级复杂文档的RAG系统。不是简单的"上传PDF然后问答"——而是直接挑战超长、多层级、跨表格与文本、且需要极高准确率的真实业务场景。这是2025年算法工程师最核心、最实战的技能栈。

**核心算法/技术**：
- **复杂文档解析**：企业级招投标文件/合同/技术规格书的解析（非结构化+半结构化）
- 智能分块策略：按语义/结构/递归/视觉布局分块（LayoutParser/Marker）
- 表格-文本联合理解：跨页表格、嵌套表格、表格与正文的引用关系
- Embedding模型选型与微调（MTEB榜单、领域适配）
- 混合检索：稠密向量检索 + 稀疏BM25 + 重排序（Rerank）
- Query改写与扩展（HyDE、多查询生成）
- 检索结果融合与排序学习
- 上下文压缩与Prompt工程

**真实业务场景**：
- **核心挑战文档**：企业级招投标文件（通常200-500页，包含技术参数、商务条款、合规要求、图纸说明）
- **难点**：
  - 同一个"产品规格"可能在文档的3个不同位置以不同格式出现（正文描述、参数表、附录图纸）
  - 需要判断"甲方要求的带宽≥10Gbps"与"乙方投标的带宽为10000Mbps"是否一致
  - 表格中的跨页数据需要自动合并
  - 某些关键条款隐藏在脚注或附件中

**AI辅助学习策略**：
- 让AI生成不同分块策略的对比实验框架，你负责跑实验和分析结果
- 用AI解释"为什么同样的Embedding模型，在不同领域表现差异巨大"
- 让AI帮你设计RAG系统的评测指标和测试集构建方案
- 阅读LangChain/LlamaIndex源码时，让AI帮你画架构图
- 让AI生成模拟的招投标文件样本（包含故意设计的歧义和矛盾），测试你的系统

**项目交付物**：
1. 一个完整的RAG系统（支持PDF/Word/Excel/Markdown/PPT上传）
2. **复杂文档解析引擎**：能处理跨页表格、嵌套结构、脚注、附录
3. 可配置的检索策略（纯向量/混合/HyDE/结构化过滤）
4. 评测框架：自动评测回答的准确性、召回率、一致性
5. **真实场景测试集**：至少3份企业级招投标文件的问答对（50+ QA pairs）
6. 技术文档：每个环节的技术选型和原因分析
7. 生产部署方案：Docker Compose + Milvus + FastAPI

**验证标准**：
- 在招投标文件测试集上，回答准确率 > 85%（包含跨段落引用）
- 表格数据提取准确率 > 90%
- 端到端延迟 < 5秒（包含文档解析+检索+生成）
- 支持并发查询（QPS > 10）
- 提供完整的消融实验报告（对比不同检索策略的效果）

**为什么用复杂文档而不用常规PDF**：
- 现实世界中的数据从来不是干净的——残缺、矛盾、格式混乱才是常态
- 能处理高价值、高复杂度业务数据的RAG系统，才是真正值钱的技能
- 面试时，"我做过200页招投标文件的跨表格+跨段落联合检索"远比"我用LangChain搭了一个PDF问答demo"有说服力

### 项目4：多Agent业务编排系统（进阶项目，4周）

**目标**：理解AI Agent的编排与调度机制，并将其应用于高价值的真实业务场景。单个LLM调用只是函数调用，多Agent协作才是真正的生产力放大器。本项目的核心挑战不是"能跑通几个Agent"，而是"编排几十个Agent实例协同处理极度非结构化的复杂业务"。

**核心算法/技术**：
- Agent架构模式：ReAct、Plan-and-Execute、Reflection
- 多Agent协作：分层架构、消息总线、任务委托
- 工具调用（Function Calling）设计与实现
- 记忆系统设计（短期/长期/语义记忆）
- 工作流编排（DAG、状态机）
- MCP（Model Context Protocol）协议理解与实现
- **大规模Agent编排**：数十个Agent实例的协同、冲突解决、状态同步

**真实业务场景——复杂采购需求拆解与合规性审查**：
- **场景描述**：收到一份企业采购需求（可能只有几行模糊描述+若干附件），需要自动完成：
  1. **需求分析Agent**：从模糊描述中提取关键技术参数
  2. **市场调研Agent**：搜索可用供应商和产品线
  3. **合规审查Agent**：对照企业采购政策和行业法规进行逐条检查
  4. **技术应答Agent**：生成技术方案的对比分析报告
  5. **商务应答Agent**：生成商务条款和风险评估
  6. **综合评审Agent**：汇总各方输出，生成最终的采购决策建议
- **难点**：每个Agent的输出都可能是非结构化的、包含不确定性的、需要后续Agent理解并处理的

**AI辅助学习策略**：
- 让AI对比不同Agent架构模式的优劣，用表格+流程图呈现
- 用AI生成一个复杂采购任务的Agent分解方案，你来评估和调整
- 让AI帮你设计Agent之间的通信协议和错误恢复机制
- 让AI充当"恶意供应商"，故意提供矛盾或不完整的数据，测试系统的鲁棒性
- 用AI模拟技术评审会议，对你的Agent编排方案提出挑战

**项目交付物**：
1. 一个多Agent协作系统（至少6种角色Agent，支持横向扩展）
2. 可视化工作流编排界面
3. Agent性能监控面板（响应时间、成功率、Token消耗、冲突率）
4. **复杂业务场景评测**：用3-5份真实采购需求测试全流程自动化
5. 案例分析文档：对比单Agent vs 多Agent在复杂任务上的表现
6. 错误恢复报告：Agent失败/超时/矛盾输出时的系统行为

**验证标准**：
- 能自主完成一个包含6+步骤的复杂采购需求拆解任务
- 全流程任务成功率 > 75%（含错误重试）
- 支持动态工具注册和发现
- 提供完整的错误处理和重试机制（单个Agent失败不阻塞整个流程）
- Agent之间的信息传递准确率 > 85%（人工抽检）

**架构思维升级**：
- Agent系统本质上是复杂分布式服务，你的架构能力在这里完全适用
- 理解"控制流 vs 数据流"在AI系统中的新含义
- 处理极度非结构化、高价值业务数据的能力，是真正值钱的专家技能

### 项目5：实时推荐引擎（进阶项目，4周）

**目标**：构建一个工业级推荐系统。推荐是算法商业价值最高的方向之一，也是经典ML与深度学习融合的典型场景。

**核心算法/技术**：
- 召回层：协同过滤（UserCF/ItemCF）、向量召回（双塔模型）
- 排序层：LR、GBDT、Wide&Deep、DeepFM
- 特征工程：用户画像、物品画像、上下文特征
- 冷启动策略：内容召回、探索利用（EE问题、Bandit算法）
- 在线学习：实时特征更新、模型增量更新
- A/B测试框架与效果评估

**AI辅助学习策略**：
- 让AI用你熟悉的电商/视频场景举例，解释"召回->粗排->精排->重排"的漏斗逻辑
- 用AI生成模拟用户行为数据，你负责构建完整的推荐pipeline
- 让AI解释Wide&Deep中"Wide学习记忆、Deep学习泛化"的直觉
- 用AI帮你设计A/B测试的实验方案（流量分配、指标选择、显著性检验）

**项目交付物**：
1. 完整推荐系统（召回+排序+重排）
2. 离线训练Pipeline（特征工程->模型训练->评估）
3. 在线服务（用户请求->实时推荐->返回结果）
4. A/B测试报告（对比不同算法的CTR/转化率）
5. 系统架构图：数据流、模型更新流、服务部署图

**验证标准**：
- 在MovieLens或模拟数据集上，Recall@50 > 20%
- 在线推荐延迟 < 200ms（P99）
- 支持新用户冷启动（首次交互后5分钟内能产生推荐）
- 提供完整的离线评测指标和在线实验报告

### 项目6：LLM推理优化与部署（深入项目，3周）

**目标**：深入理解LLM的推理机制，掌握模型部署优化的核心技术。不仅会调vLLM，更要理解数据从磁盘到显存/统一内存的每一个流转环节。这是区分"调包侠"与"算法工程师"的关键分水岭。

**核心算法/技术**：
- Transformer架构深入：自注意力机制、多头注意力、位置编码
- KV Cache原理与优化（PagedAttention、vLLM）
- 量化技术：INT8/INT4/AWQ/GPTQ的原理与效果
- **GGUF格式深入**：量化方案的元数据组织、内存布局、加载流程
- 推测解码（Speculative Decoding）加速原理
- 连续批处理（Continuous Batching）vs 静态批处理
- 模型并行：张量并行、流水线并行
- **本地优先（Local-First）架构**：
  - Apple Silicon（M1/M2/M3/M4）统一内存架构下的模型加载与推理优化
  - MLX框架：原生Apple芯片推理引擎的使用与优化
  - Thunderbolt级别极速SSD下的I/O策略：模型分页加载、懒加载、预取
  - 内存带宽 vs 显存带宽的性能差异分析

**AI辅助学习策略**：
- 让AI用图解方式解释KV Cache"为什么能避免重复计算"
- 用AI对比不同量化方案的精度损失和速度提升，画决策树
- 让AI帮你分析vLLM源码中Scheduler的核心逻辑
- 让AI生成不同batch size下的延迟-吞吐量曲线分析
- **新增**：让AI帮你设计Apple Silicon vs NVIDIA GPU的性能对比实验

**项目交付物**：
1. 一个基于vLLM的模型推理服务（支持Qwen/Llama系列）
2. **本地优先方案**：同一模型在Apple Silicon（MLX/GGUF）和NVIDIA GPU（vLLM）下的完整性能对比报告
3. 自动量化Pipeline（FP16->INT8/INT4/GGUF Q4_K_M/Q5_K_M）
4. 推理服务监控面板（Token吞吐量、延迟分布、GPU/内存利用率、I/O等待时间）
5. **I/O性能分析报告**：模型加载过程中的磁盘读、内存分配、带宽利用率
6. 技术博客：《LLM推理优化的工程实践：从云端到本地》

**验证标准**：
- 单卡A100上，7B模型的推理吞吐量 > 1000 tokens/s
- Apple Silicon（M2/M3 24GB+）上，7B模型的推理吞吐量 > 40 tokens/s
- 首Token延迟 < 500ms（输入长度 < 2K）
- 支持动态Batch和流式输出
- 量化后模型在标准评测集上的精度损失 < 5%
- GGUF模型从磁盘加载到可推理的时间 < 15秒（Thunderbolt SSD）

**为什么要深入底层I/O和内存**：
- 专家必须理解数据流的每一步：模型权重从磁盘→内存→芯片统一内存→计算单元的路径
- Local-First正在成为趋势（隐私、成本、延迟），不能只依赖云端A100
- 面试时，"我分析过MLX在Apple Silicon上的内存访问模式和cache miss率"是碾压级的回答

### 项目7：领域模型微调（SFT/RLHF）（深入项目，4周）

**目标**：掌握模型微调的核心技术栈。Prompt Engineering和RAG无法解决根本的领域认知问题——当特定领域的私有知识需要被"烤"进模型权重时，你必须会微调。这是专家不可或缺的底层手感。

**核心算法/技术**：
- 监督微调（SFT）：全参数微调 vs 参数高效微调（PEFT）
- LoRA/QLoRA：低秩适配的原理、rank选择、alpha配置、量化-aware微调
- 数据配比策略：高质量数据 vs 大数据量、通用数据 vs 领域数据的混合比例
- Loss曲线解读：收敛规律、过拟合信号、学习率调优
- **灾难性遗忘的防止**：混合训练、正则化（EWC）、回放策略
- RLHF（Reinforcement Learning from Human Feedback）：奖励模型训练、PPO/DPO对齐
- 评估体系：自动化评测（BLEU/ROUGE/LLM-as-Judge）+ 人工评估设计

**实战化改造——领域知识注入**：
- **不去微调写诗或闲聊的模型**。选择以下真实场景之一：
  - **代码AST规则注入**：将项目1中的代码审查规则"烤"进模型，让模型学会特定团队的代码规范
  - **招投标文档理解**：让模型学会特定行业（如IT基础设施采购、医疗设备采购）的术语、标准和合规要求
  - **企业内部知识**：将某个开源项目的架构决策、编码规范、API文档"烤"进7B模型
- **核心指标**：微调后的模型在领域任务上的表现 > GPT-4o水平，同时通用能力下降 < 10%

**AI辅助学习策略**：
- 让AI帮你设计数据配比实验（10%领域+90%通用 vs 30%领域+70%通用 vs 50%领域+50%通用）
- 用AI解释LoRA中rank和alpha的选择对效果的影响直觉
- 让AI生成"灾难性遗忘"的诊断报告，帮你找到遗忘的模式
- 用AI对比不同对齐方法（RLHF vs DPO vs ORPO）的优劣
- 让AI帮你设计自动评估pipeline（避免纯人工打分）

**项目交付物**：
1. 完整的微调Pipeline（数据准备→训练→评估→导出）
2. LoRA/QLoRA微调的7B领域模型（可部署推理）
3. **数据配比实验报告**：不同配比下的领域能力 vs 通用能力trade-off曲线
4. **Loss曲线分析文档**：至少3次不同超参的训练记录，标注关键转折点
5. 灾难性遗忘评估报告（训练前后的通用能力对比）
6. 模型部署方案（GGUF量化后可本地推理）
7. 技术博客：《把领域知识烤进模型：LoRA微调的实战指南》

**验证标准**：
- 在领域测试集上，微调模型的准确率超过同尺寸基线模型 > 30%
- 通用能力（MMLU/CMMLU子集）下降 < 10%
- LoRA adapter文件大小 < 500MB（7B模型）
- 提供至少3组不同超参的对比实验
- 模型可通过GGUF量化部署到Apple Silicon本地推理

### 项目8：多模态创意生成工作流（深入项目，4周）

**目标**：掌握多模态AI的核心技术栈。在2026年的AI生态中，缺乏多模态处理能力会极大限制技术视野。本项目探索Vision-Language Models（VLM）和图像生成模型的结合，构建端到端的创意生成管线。

**核心算法/技术**：
- Vision-Language Models（VLM）：CLIP、LLaVA、Qwen-VL的架构与使用
- 图像反推Prompt：从真实照片反生成高质量文本描述（BLIP-2 / LLaVA-Caption）
- 图像生成模型：Stable Diffusion XL / Flux / Midjourney API的原理与使用
- ControlNet：姿态、边缘、深度图等多条件约束下的精准生成
- 风格迁移与一致性：特定文化主题（如《山海经》风格）的模型适配
- 多模态RAG：结合图像检索+文本检索的混合搜索
- 图像质量评估：FID、CLIP Score、人工评估设计

**真实业务场景——现实照片到游戏素材的端到端转换**：
- **输入**：一张真实的现实世界照片（如风景照、建筑照片、人物照片）
- **输出**：具有特定文化主题（如《山海经》风格）的创意游戏内素材
- **流程**：
  1. **图像理解**：VLM分析原照片的构图、主体、光线、氛围
  2. **Prompt反推**：生成结构化的生成Prompt（包含主题、风格、元素描述）
  3. **风格约束**：通过ControlNet/LoRA注入特定文化风格的视觉元素
  4. **生成与迭代**：多轮生成+VLM评估反馈（生成→评分→调整Prompt→再生成）
  5. **后期处理**：分辨率增强、风格统一、多视角一致性

**AI辅助学习策略**：
- 让AI用图解方式解释Stable Diffusion的扩散过程和UNet架构
- 用AI分析不同ControlNet预处理器（Canny/Depth/Pose）对生成结果的影响
- 让AI帮你设计"图像质量自动评估"的pipeline（VLM-as-Judge）
- 用AI生成不同文化风格的数据集样本，用于训练风格LoRA
- 让AI对比ComfyUI vs Diffusers vs Midjourney API的工程优劣

**项目交付物**：
1. 完整的多模态生成管线（输入照片→分析→生成→迭代→输出）
2. 特定文化主题的风格LoRA（训练好的adapter）
3. 图像质量自动评估系统（VLM-based scoring）
4. 生成效果对比报告（输入/输出对比、风格一致性评估、生成成功率）
5. ComfyUI工作流文件（可复用的节点图）
6. 技术博客：《从现实到幻想：多模态AI创意生成的工程实践》

**验证标准**：
- 图像风格一致性评分（CLIP Score）> 0.75
- 生成成功率 > 70%（3次迭代内生成满意结果）
- 单次完整流程延迟 < 30秒（含生成+迭代）
- 提供至少20组输入/输出对比案例
- 风格LoRA文件大小 < 300MB

**目标**：综合运用前8个项目的能力，设计一个企业级AI中台。这不是写代码的项目，而是架构设计+原型验证。需要整合文本生成、向量检索、推荐、多模态生成、微调模型等多种AI能力。

## 给工程师的特别提醒

1. **你的工程背景是最大优势**：不要觉得"不懂数学"是障碍。工程直觉（系统设计、性能优化、代码质量）是很多纯算法背景的人缺乏的。但也不要停留在"会调API"——专家和普通开发者的分水岭在于底层理解。

2. **不要陷入"先补基础"的陷阱**：传统路线要求先学线性代数、概率论、微积分，再学ML。项目制路线的核心是"在做中学"——项目中需要什么，就学什么。但在微调项目（项目7）中，你至少需要理解梯度下降、学习率、过拟合的基本概念。

3. **算法不是目的，解决问题才是**：每个算法选择都应该有工程理由。面试时，"为什么选这个算法"比"这个算法怎么实现"更重要。但如果面试官追问"数据在内存中怎么存的"，你也要能答上来。

4. **保持架构师视角**：即使是学习算法，也要思考"这个算法在系统中处于什么位置""瓶颈在哪里""如何扩展"。从项目2开始，每个项目都至少做一次profiling分析（CPU cache、内存带宽、I/O延迟）。

5. **多模态不是选修课**：2026年的AI生态已经不可能只处理纯文本。项目8的多模态能力（VLM理解+图像生成）将是你区别于"纯NLP算法工程师"的重要差异化技能。

6. **微调是专家的必修课**：RAG和Prompt Engineering能解决70%的场景，但剩下的30%——那些需要模型真正"理解"领域知识的场景——必须靠微调。项目7的数据配比实验和Loss曲线分析，是你建立"模型手感"的关键。

7. **9个月后的你**：拥有9个完整项目、扎实的技术博客、可以直接展示的GitHub作品集。你的能力栈覆盖：文本理解、向量检索、RAG、Agent编排、推荐系统、推理优化、模型微调、多模态生成、架构设计。这时候你已经具备了算法专家的实战能力，剩下的只是在工作中继续深化。
